# Pareto LLM Benchmark Suite â€” Implementation Plan

## Goal

Build a CLI tool that systematically benchmarks local LLMs across model variants and
quantization levels, capturing raw performance and quality data so the **Pareto frontier**
can be plotted along any dimensions the user chooses after the fact.

---

## Assumed Stack

| Concern | Choice | Notes |
|---|---|---|
| LLM backend (Mac) | **MLX** (Apple Silicon) | Via `mlx-lm`; uses Metal for GPU inference |
| LLM backend (Linux) | **llama.cpp** | Via `llama-cpp-python`; requires NVIDIA GPU + CUDA |
| Model registry | Hugging Face Hub | MLX-Community models (Mac), GGUF models (Linux) |
| GPU monitoring (Mac) | `mlx.core.metal` | Metal memory via `get_active_memory()` |
| GPU monitoring (Linux) | `pynvml` | NVIDIA GPU memory via NVML bindings |
| Result storage | **CSV** | One row per individual run; simple, portable, no query overhead |
| Config format | **YAML** | Human-friendly |
| CLI | `click` | Clean subcommand support |
| Python version | 3.11+ | Match MLX requirements |
| Package manager | `uv` | Fast, lockfile-based |
| Settings | `.env` file (not committed) | Machine-local params; loaded via `python-dotenv` |

## Environment / Params File

### `scripts/init_env.py`

Generates a `.env` file in the repo root on first run (or re-run to reset). The file is
**never committed** (added to `.gitignore`).

```python
# Pseudocode for init_env.py
import platform, pathlib, shutil, subprocess

system = platform.system()

def detect_gpu_backend() -> str:
    if system == "Darwin":
        return "mlx"
    # Linux: check for NVIDIA GPU
    if shutil.which("nvidia-smi"):
        try:
            subprocess.run(["nvidia-smi"], capture_output=True, check=True)
            return "cuda"
        except subprocess.CalledProcessError:
            pass
    raise RuntimeError("No supported GPU found. Requires Apple Silicon (Metal) or NVIDIA (CUDA).")

gpu_backend = detect_gpu_backend()

env_content = f"""
# Auto-generated by scripts/init_env.py â€” do not commit
GPU_BACKEND={gpu_backend}
RESULTS_DIR=./results
KEEP_MODEL_FILES=false
""".strip()

pathlib.Path(".env").write_text(env_content)
print(f"Wrote .env  (GPU_BACKEND={gpu_backend})")
```

- `GPU_BACKEND` is one of `mlx` (Mac/Metal) or `cuda` (Linux/NVIDIA). Determines which
  backend is used for inference and which GPU memory sampling strategy is active.
- Future: `rocm` for AMD GPUs on Linux (not yet implemented).
- All other tuneable machine-local values (results output dir, cache dir, etc.) also live here.
- The main application loads `.env` at startup via `python-dotenv` before reading config.

## Module Specifications

### `backend/base.py`

```python
@dataclass
class GenerationResult:
    text: str
    prompt_tokens: int
    gen_tokens: int
    prompt_tps: float   # prompt processing tokens/sec
    gen_tps: float      # generation tokens/sec

class LLMBackend(ABC):
    @abstractmethod
    def load(self, model_id: str) -> None: ...

    @abstractmethod
    def generate(self, prompt: str, max_tokens: int = 512) -> GenerationResult: ...

    @abstractmethod
    def unload(self) -> None: ...

    @abstractmethod
    def max_context_tokens(self) -> int: ...
```

### `backend/llamacpp_backend.py`

Implements `LLMBackend` for Linux using `llama-cpp-python` (Python bindings for llama.cpp).

Key details:
- **Model format**: GGUF files from Hugging Face Hub. Model IDs use the format
  `repo_id:filename_pattern` (e.g., `bartowski/Qwen2.5-7B-Instruct-GGUF:Q4_K_M`).
  The filename pattern is matched against available GGUF files in the repo.
- **GPU offloading**: All layers offloaded to NVIDIA GPU by default (`n_gpu_layers=-1`).
- **Model download**: Uses `huggingface_hub.hf_hub_download()` to fetch the GGUF file.
- **`load(model_id)`**: Downloads GGUF if not cached, instantiates `llama_cpp.Llama`.
- **`generate(prompt, max_tokens)`**: Calls `llama.create_completion()`, extracts timing
  info from the `timings` field in the response to populate `GenerationResult`.
- **`unload()`**: Deletes the `Llama` instance to free VRAM.
- **`max_context_tokens()`**: Returns `llama.n_ctx()`.

### `benchmarks/base.py` â€” Plugin System

Benchmarks are **self-registering** via a class decorator so new ones can be added as
modules without touching any central registry file:

```python
BENCHMARK_REGISTRY: dict[str, type[Benchmark]] = {}

def register(key: str):
    def decorator(cls):
        BENCHMARK_REGISTRY[key] = cls
        return cls
    return decorator

@dataclass
class BenchmarkResult:
    score: float           # 0.0â€“1.0; benchmark-defined meaning
    extra: dict            # arbitrary benchmark-specific data serialised into CSV

class Benchmark(ABC):
    def __init__(self, config: dict) -> None: ...

    @abstractmethod
    def run_single(self, backend: LLMBackend) -> BenchmarkResult: ...
```

Usage:

```python
# benchmarks/coding.py
@register("coding")
class CodingBenchmark(Benchmark): ...
```

The runner discovers all benchmark modules by importing `pareto_llm.benchmarks.*` at
startup (or via explicit import in `__init__.py`), which triggers registration.

### Context Length Benchmarks

The `context_length` benchmark type is a **wrapper** around any inner benchmark.
Before running the inner benchmark, it prepends padding text to each prompt so the
model's context is at the target fill ratio when the real question begins.

```
final_prompt = [padding_to_fill_ratio] + [inner_benchmark_prompt]
```

Key considerations:
- Padding token count = `int(backend.max_context_tokens() * fill_ratio) - len(inner_prompt_tokens) - safety_margin`
- Padding source: public-domain text (Project Gutenberg), truncated/repeated to hit
  the exact token budget. The backend tokenizes the padding to verify length.
- The inner benchmark's `run_single` is called with a modified backend wrapper that
  injects the prefix transparently â€” the inner benchmark doesn't need to know about it.
- Each inner benchmark may need its own padding strategy if its prompts are structured
  (e.g. chat turns vs. raw completion). This is handled by an optional
  `supports_context_padding: bool = True` flag on `Benchmark` subclasses. If `False`,
  the context wrapper raises a clear error at config-parse time.

### `metrics/system.py`

Collects system metrics during a single generation call:

```python
@dataclass
class SystemMetrics:
    ram_max_gb: float
    ram_avg_gb: float
    gpu_ram_max_gb: float | None
    gpu_ram_avg_gb: float | None
    # T/s and PP/s come directly from GenerationResult â€” no polling needed
```

- RAM sampled in a background thread (10Hz) using `psutil.Process().memory_info().rss`
- GPU memory sampling depends on `GPU_BACKEND`:
  - `mlx`: `mlx.core.metal.get_active_memory()` for Metal GPU RAM
  - `cuda`: `pynvml.nvmlDeviceGetMemoryInfo()` for NVIDIA GPU RAM
  - Future `rocm`: not yet implemented
- GPU fields stored as `gpu_ram_max_gb` / `gpu_ram_avg_gb` columns
- Implemented as a context manager: `with SystemMetricsCollector(gpu_backend=...) as m: ...`

### `storage/csv_writer.py`

Appends one row per completed run to a CSV file. The CSV is **append-only** during a
run; if the file doesn't exist it is created with a header row.

#### CSV Columns

| Column | Source |
|---|---|
| `timestamp` | ISO-8601 wall time at start of run |
| `run_label` | From config `run_label` |
| `model_id` | Full HF repo ID |
| `benchmark_name` | From config `benchmarks[].name` |
| `run_num` | 1-based counter within this model+benchmark combination |
| `score` | `BenchmarkResult.score` |
| `prompt_tokens` | `GenerationResult.prompt_tokens` |
| `gen_tokens` | `GenerationResult.gen_tokens` |
| `gen_tps` | `GenerationResult.gen_tps` |
| `prompt_tps` | `GenerationResult.prompt_tps` |
| `ram_max_gb` | `SystemMetrics.ram_max_gb` |
| `ram_avg_gb` | `SystemMetrics.ram_avg_gb` |
| `gpu_ram_max_gb` | Metal (Mac) or NVIDIA (Linux); `null` if unavailable |
| `gpu_ram_avg_gb` | Metal (Mac) or NVIDIA (Linux); `null` if unavailable |
| `max_ctx_tokens` | `backend.max_context_tokens()` |
| `extra_*` | Flattened `BenchmarkResult.extra` dict (prefixed with `extra_`) |

No aggregation is performed. Each row is one raw run. Downstream analysis (averaging,
Pareto plotting) happens outside this tool.

## Runner Flow (`runner.py`)

```
load .env
parse config yaml â†’ BenchmarkConfig
resolve benchmark classes from registry
backend = create_backend(gpu_backend)   # "mlx" â†’ MLXBackend, "cuda" â†’ LlamaCppBackend

for model_id in config.models:
    backend.load(model_id)
    for benchmark_entry in config.benchmarks:
        benchmark = BENCHMARK_REGISTRY[benchmark_entry.type](benchmark_entry.config)
        for run_num in range(1, defaults.runs_per_test + 1):
            try:
                with SystemMetricsCollector() as metrics:
                    result = benchmark.run_single(backend)
                csv_writer.append(model_id, benchmark_entry.name, run_num, result, metrics, backend)
            except Exception as e:
                log.error(f"[{model_id}] [{benchmark_entry.name}] run {run_num} failed: {e}")
                csv_writer.append_failure(model_id, benchmark_entry.name, run_num, error=e)
    backend.unload()
    if not keep_model_files:
        delete_hf_cache(model_id)
```

## Test-Driven Development Plan

All tests are written **before** the implementation code they cover.
Use `pytest` with `pytest-cov`. Backend-specific tests gated with markers (e.g.,
`pytest.mark.mlx` skipped when `GPU_BACKEND != mlx`, `pytest.mark.cuda` skipped when
`GPU_BACKEND != cuda`).

### Phase 0 â€” Scaffolding
- [ ] `pyproject.toml` with `[project.optional-dependencies] dev = [pytest, pytest-cov, ruff, mypy]`
- [ ] `conftest.py` fixtures: `tmp_csv_path`, `mock_backend`, `sample_config`, `mock_benchmark`
- [ ] `pytest.ini` markers: `mlx`, `cuda`, `integration`

### Phase 1 â€” Environment init (`test_env.py`)
- `init_env.py` creates `.env` when it doesn't exist
- `GPU_BACKEND=mlx` when `platform.system() == "Darwin"` (mock platform)
- `GPU_BACKEND=cuda` when on Linux with `nvidia-smi` available (mock subprocess)
- Raises `RuntimeError` when no supported GPU is detected
- Re-running overwrites cleanly
- `.env` is present in `.gitignore`

### Phase 2 â€” Config (`test_config.py`)
- Parse valid YAML â†’ `BenchmarkConfig`
- Validation errors: unknown benchmark type, missing `run_label`, invalid `fill_ratio`
- `BenchmarkEntry.config` is passed through as a raw dict (benchmark owns its schema)

### Phase 3 â€” Benchmark registry (`test_benchmarks.py`)
- `@register("foo")` adds class to `BENCHMARK_REGISTRY`
- Duplicate key raises at import time
- `BENCHMARK_REGISTRY["foo"](config).run_single(mock_backend)` returns `BenchmarkResult`
- Context-length wrapper: padding injected, inner benchmark called with padded prompt
- Context-length wrapper: raises if inner benchmark has `supports_context_padding = False`

### Phase 4 â€” Metrics (`test_metrics.py`)
- `SystemMetricsCollector` returns correct `ram_max_gb` and `ram_avg_gb` over a mock workload
- GPU fields populated when `GPU_BACKEND=mlx` (mock `mlx.core.metal`) or `GPU_BACKEND=cuda` (mock `pynvml`)
- GPU fields are `None` when GPU monitoring is unavailable
- Thread safety: context manager completes cleanly even if body raises

### Phase 5 â€” Storage (`test_storage.py`)
- `csv_writer.append(...)` creates file with header on first call
- Second call appends a row (no duplicate header)
- `extra` dict is flattened to `extra_*` columns; mismatched keys between rows fill with empty string
- All expected columns present in output

### Phase 6 â€” Runner (`test_runner.py`)
- Mock backend + two mock benchmarks + two models â†’ correct number of CSV rows
- `run_num` increments correctly within each model+benchmark block
- `backend.unload()` called once per model
- `delete_hf_cache` called when `keep_model_files=False`, not called when `True`
- Exception in a benchmark logs the error and continues to the next benchmark/model; remaining runs are not skipped
- Failed rows are written to CSV with `score=null` and an `extra_error` field containing the exception message

### Phase 7 â€” MLX backend (`test_mlx_backend.py`) â€” `@pytest.mark.mlx`
- Requires Apple Silicon; skipped when `GPU_BACKEND != mlx`
- `load` + `generate` + `unload` with a small real model
- `max_context_tokens()` returns a positive integer

### Phase 8 â€” llama.cpp backend (`test_llamacpp_backend.py`) â€” `@pytest.mark.cuda`
- Requires Linux + NVIDIA GPU; skipped when `GPU_BACKEND != cuda`
- `load` with a GGUF model from HF Hub + `generate` + `unload`
- `max_context_tokens()` returns a positive integer

## Out of Scope (for now)

- KV quantization testing (noted in original spec as "probably too in depth")
- GUI / dashboard â€” raw CSV is the output; plotting is external
- Distributed/parallel model testing (sequential required for accurate RAM measurement)
- AMD ROCm GPU support (abstraction layer makes this a future add-on)
- vLLM backend (abstraction layer makes this a future add-on)
- CPU-only inference (GPU required for both backends)
- Pareto computation â€” will be done in a separate downstream process
